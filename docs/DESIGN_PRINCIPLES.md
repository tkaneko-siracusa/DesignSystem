# Polastack Design System - デザイン原則

> 本ドキュメントは、デザインシステム全体を貫く判断基準を定義する。
> 全コンポーネント・パターンの設計はこの原則に基づく。

---

## デザイン原則

### 1. 明快さ（Clarity）

**情報の優先順位が一目で分かるUIを作る。**

- 視覚的な階層を明確にする（サイズ、色、余白で重要度を表現）
- ラベルやアクションの意味が推測なしに理解できること
- アイコンだけに頼らず、テキストラベルを併用する
- 状態変化（ローディング、エラー、成功）を即座に視覚フィードバックする

### 2. 一貫性（Consistency）

**同じ操作は、どのアプリでも同じ結果を生む。**

- 同じ機能には同じコンポーネントを使用する（独自実装を避ける）
- 用語・ラベルをプラットフォーム全体で統一する
- インタラクションパターン（クリック、ホバー、キーボード操作）を標準化する
- 例外を作る場合は、明確な理由を文書化する

### 3. 効率性（Efficiency）

**ユーザーが最小の操作で目的を達成できるUIを作る。**

- 頻繁に使う操作は最短の動線で到達できること
- 一覧→詳細→編集の動線を画面遷移なしで完結（Drawer中心パターン）
- キーボードショートカットを重要な操作に提供する
- バルク操作（一括選択・一括編集・一括削除）を標準パターンとして提供する

### 4. 適応的密度（Adaptive Density）

**デバイスとコンテキストに応じて、情報密度を最適化する。**

- デスクトップ：高密度レイアウトで一画面の情報量を最大化
- タブレット：適度な余白でタッチ操作を可能に
- モバイル：重要な情報に絞り、段階的に開示
- 同じコンポーネントが、コンテキストに応じて適切な密度で表示されること

### 5. 段階的開示（Progressive Disclosure）

**必要な情報を、必要なタイミングで表示する。**

- 初期表示では最も重要な情報のみ表示
- 詳細情報はユーザーのアクション（クリック、展開）で表示
- 高度な機能は初心者を圧倒しないよう、段階的にアクセス可能にする
- ツールチップ、ヘルプテキスト、インラインドキュメントで必要な時だけ補足

## 適用ガイドライン

### デザインレビューチェックリスト

コンポーネントや画面を設計・レビューする際に確認する：

- [ ] **明快さ**: 初めて見るユーザーが、何をすべきか3秒以内に理解できるか？
- [ ] **一貫性**: 既存のコンポーネントやパターンと矛盾していないか？
- [ ] **効率性**: この操作を完了するのに不必要なステップがないか？
- [ ] **適応的密度**: デスクトップとモバイルの両方で適切に表示されるか？
- [ ] **段階的開示**: 初期表示の情報量は適切か？詳細は必要に応じて見られるか？

### 原則の優先順位

原則が互いに矛盾する場合：

1. **明快さ** > 効率性（効率を追求しても、理解しにくければ本末転倒）
2. **一貫性** > 個別最適化（特殊対応より、統一パターンの維持を優先）
3. **効率性** > 密度（密度を上げるより、操作ステップの削減を優先）
